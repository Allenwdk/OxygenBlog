## 编译llama.cpp

llamacpp是LM Studio的后端，但是他并没有如LM Studio一般的前端界面和安装程序。要想达到性能利用最大化，你需要在自己的设备上手动编译。

1. 打开“终端”，在顶部下拉菜单中选择`Developer Powershell For VS 18`
2. 克隆源码

   llama.cpp官方：[ggml-org/llama.cpp: LLM inference in C/C++](https://github.com/ggml-org/llama.cpp)
   ```PowerShell
   git clone https://github.com/ggml-org/llama.cpp.git
   cd llama.cpp
   ```
   或buun-llama-cpp（支持Turbo Quant）：[spiritbuun/buun-llama-cpp: LLAMA Turboquant implementation with CUDA support](https://github.com/spiritbuun/buun-llama-cpp)
   ```PowerShell
   git clone https://github.com/spiritbuun/buun-llama-cpp.git
   cd buun-llama-cpp
   ```
3. 开始编译
   ```PowerShell
   cmake -B build -G "Ninja" `
    -DGGML_CUDA=ON `
    -DGGML_NATIVE=ON `
    -DGGML_CUDA_FA=ON `
    -DGGML_CUDA_FA_ALL_QUANTS=ON `
    -DCMAKE_BUILD_TYPE=Release `
    -DCMAKE_CUDA_FLAGS="-Xcompiler /utf-8" `
    -DCMAKE_CXX_FLAGS="/utf-8" `
    -DCMAKE_C_FLAGS="/utf-8"
   ```
   ```PowerShell
   cmake --build build --parallel 32
   ```
   如果没有报错，那就说明成功了（）

## 启动参数

现在才是重头戏。接下来的大部分内容我会围绕Qwen3.6-35B-A3B-APEX-GGUF展开，因为我一直在调这个模型（）

### 模型路径

- 使用 `-m` 指定模型gguf文件路径
- 使用 `--mmproj` 指定模型视觉文件路径

建议将路径用双引号包裹，防止空格、转义问题

### 端口及地址

使用 `--port <端口号>` 来指定api端口，默认端口我忘记了（）

使用 `--host` 设置api的地址，如果仅在当前设备上使用则无需设置，如果要在局域网使用可设置为 `0.0.0.0`

### 基础参数

根据官方说法，

> **We recommend using the following set of sampling parameters for generation**
>
> - \*\*Thinking mode for general tasks: \*\*`temperature=1.0, top_p=0.95, top_k=20, min_p=0.0, presence_penalty=1.5, repetition_penalty=1.0`
> - \*\*Thinking mode for precise coding tasks (e.g. WebDev): \*\*`temperature=0.6, top_p=0.95, top_k=20, min_p=0.0, presence_penalty=0.0, repetition_penalty=1.0`
> - \*\*Instruct (or non-thinking) mode for general tasks: \*\*`temperature=0.7, top_p=0.8, top_k=20, min_p=0.0, presence_penalty=1.5, repetition_penalty=1.0`
> - \*\*Instruct (or non-thinking) mode for reasoning tasks: \*\*`temperature=1.0, top_p=0.95, top_k=20, min_p=0.0, presence_penalty=1.5, repetition_penalty=1.0`
>
> **Please note that the support for sampling parameters varies according to inference frameworks.**

所以在编程场景和日用场景，我均选择`temperature=1.0, top_p=0.95, top_k=20, min_p=0.0, presence_penalty=1.5, repetition_penalty=1.0`

在llama.cpp中即

```PowerShell
--temp 1 `
--top-p 0.95 `
--top-k 20 `
--min-p 0.00 `
--presence-penalty 1.5 `
--repeat-penalty 1.0 `
```

### 模型名称

使用 `--alias <名称>` 参数即可指定模型在调用时的名称，建议设的简单一些并固定以方便配置

### 上下文长度

使用 `--ctx-size <上下文长度>` 来指定上下文长度。通常建议设置262144（模型训练最大值）/256000/128000以便装下较长的系统提示词

### KV缓存量化

**重头戏来辣**

kv缓存，可以理解为模型的上下文。通常一个模型在显存中上下文的占用会随着上下文长度线性增长，256k的上下文需占用10G+显存。这时kv缓存量化就起到了重要的作用。它就像大模型的量化一样，可以将上下文占用显著减少，并且几乎不影响速度和性能。

llama.cpp中常见的kv量化方式有以下几种：

`F16 BF16 q8_0 q5_1 q5_0 q4_1 q4_0`  

他们的精度从左到右递减，显存占用从左到右也是递减。通常使用q8\_0是比较好的平衡

而前文提到的Turbo Quant提供了两种新的量化方式，`turbo4` 和 `turbo3`。通过一些奇妙的算法，使turbo4能达到近似F16（也就是约等于不量化）的精度下实现了q4\_0的显存占用，而turbo3在精度没比4低多少的情况下显存占用更低。这在小显存设备上是非常有用的

说了这么多，接下来开始我推荐的kv量化配置：

```PowerShell
-ctk turbo4
-ctv turbo4

# 或
-ctk turbo3_tcq
-ctv turbo3_tcq
# turbo3_tcq是buun作者合并的比turbo3精度更高，但是显存占用几乎不变的量化方式
```

### batch size

这个参数是用来设置预填充时批处理的个数。增加他可以提高预填充（从第一次收到请求到输出首个token）的速度

`-b` 代表一批的个数

`-ub` 代表一次处理的个数（这两处解释的一点也不精确（）只是大概意思，具体可以AI）

注意：设置-b通常不会增加显存占用，但-ub会显著增加显存占用

这两个值通常设为2的n次方，如512、1024、2048

-b通常设为-ub的2倍

我正在使用的参数是：

```PowerShell
-b 4096
-ub 2048
```

### MOE架构模型显存优化相关参数

```PowerShell
-ngl 999 `    #将模型所有层放在显存
--cont-batching `    #玄学优化多线程process prompt速度
--n-cpu-moe 37 `    #将37个专家放在内存里，剩下的在显存
-fit off `    #关掉llama自带的自动调节
-fit-target 250 `    #保留250M显存空余，防止后续对话时爆显存
--no-mmap `    #一次性分配内存，加上这个命令你会发现共享GPU内存占了很多，会提升一些速度
--kv-unified `    #貌似没啥用
--no-context-shift    #玄学优化多线程process prompt速度
```

### 快速注意力

使用 `--flash_attn on` 手动开启快速注意力，可以加快提示词处理速度，不影响精度。

### preserve\_thinking

Qwen3.6的新功能，可以把思考过程也放在上下文里，可以提升模型性能

通过 `--chat-template-kwargs '{"preserve_thinking": true}'` 开启。不过目前没仔细测试过是否生效 就当他生效了叭（）

### 其他参数

`--offline` 可以不让llama在加载模型时访问huggingface获取信息，加快加载速度

`--threads <线程数>` 设置并行线程数，建议设为cpu核心数量

前置的环境变量：`$env:GGML_OP_OFFLOAD_MIN_BATCH="512"` 或许可以提升提示词处理速度，放在命令最前面

<br />

将以上所有参数合在一起，就可以得到完整的启动参数：

```PowerShell
$env:GGML_OP_OFFLOAD_MIN_BATCH="512"; .\llama-server `
 -m "C:\Users\wdkal\models\Qwen3.6-35B-A3B-APEX-GGUF\Qwen3.6-35B-A3B-APEX-I-Compact.gguf" `
 --mmproj "C:\Users\wdkal\models\Qwen3.6-35B-A3B-APEX-GGUF\mmproj.gguf" `
 --alias "Qwen3.6" `
 --ctx-size 256000 `
 --port 8033 `
 --fit-target 250 `
 -ngl 999 `
 --offline `
 --flash_attn on `
 --threads 32 `
 --cont-batching `
 --n-cpu-moe 32 `
 -b 4096 `
 -ub 2048 `
 -fit off `
 --host 0.0.0.0 `
 --no-mmap `
 --kv-unified `
 --chat-template-kwargs '{"preserve_thinking": true}' `
 --temp 1 `
 -ctk turbo4 `
 -ctv turbo4 `
 --top-p 0.95 `
 --top-k 20 `
 --min-p 0.00 `
 --presence-penalty 1.5 `
 --repeat-penalty 1.0 `
 --no-context-shift
```

注：执行启动命令前请先使用 `cd \build\bin\` 定位到编译出产物目录。建议切换到Powershell中运行。使用时请自行替换模型和多模态的文件位置，并根据显存大小设置--n-cpu-moe和-ub的值

<br />

### 有关MTP

在使用支持MTP的模型时，可以使用 `--parallel 1 -spec-type mtp --spec-draft-n-max 3` 来激活MTP。MTP可以一定程度的加快模型生成token速度。

## 写在最后

本地运行的小模型速度和生成质量终归不如官方的大模型。我看到网上很多评论说“低于Q6量化的模型没法用来编程”，不过我认为APEX量化提供的精度不低。这篇文章并没有结束，我会在有时间的时候补全完整的环境配置和注意事项，还有使用yarn扩展上下文的参数，或许还会再添加一些专有名词、概念的科普。如果要使用LM Studio，这篇文章的内容大部分也是适用的，只是需要在模型页面的Load和Interface下找到对应的设置项。

如果你遇到了一开始能够正常加载模型，干了会儿别的发现再加载oom了，建议重启电脑，这可能是其他软件使用显卡时残留的垃圾数据导致的。目前不太清楚不重启有没有解决方法，而且这时候查看显存也是没有占用的。S
